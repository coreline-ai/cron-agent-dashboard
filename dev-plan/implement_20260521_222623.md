## 개발 목적

backlog 순차 진행 — 네 번째 항목 **per-run worktree**. 현재 `ClaimNextRun` SQL이 `workspace_id`당 동시 running run을 1개로 제한해 한 워크스페이스 안에서는 항상 직렬 실행이다. RFP / hub-PM 워크플로우는 main agent 재진입을 빼면 깊이 8 까지 풀체인이 가능한데, 워크스페이스가 1개의 워커풀 슬롯만 점유하다 보니 다른 이슈는 큐에서 묶여 있다. 본 plan은 **workspace.per_run_worktree opt-in**을 도입해, 활성화된 워크스페이스의 각 run이 자기만의 파일 시스템 cwd(`<data_dir>/worktrees/<workspace>/<run-id>`)에서 실행되고 동시 claim이 허용되도록 한다.

## 개발 범위

### Phase 1 — schema + model
- 새 migration `0018_workspace_per_run_worktree.sql` — `workspace.per_run_worktree INTEGER NOT NULL DEFAULT 0`.
- `internal/store/models.go::Workspace`/`CreateWorkspaceInput`/`UpdateWorkspaceInput`에 `PerRunWorktree` 필드.
- `internal/store/workspace.go::CreateWorkspace` / `UpdateWorkspace` 가 새 필드 INSERT/UPDATE.

### Phase 2 — worktree manager
- `internal/app/worktree.go` 신설 — `AllocateRunWorktree(ctx, dataDir, workspaceSlug, runID) (path, cleanup func(), error)`.
- 단순 파일 시스템 격리만 본 plan에 — `<data_dir>/worktrees/<workspace-slug>/<run-id>/` 디렉토리 생성, 0700 권한.
- run 종료 시 cleanup으로 디렉토리 제거(cap을 두지 않음; tmpfs 위에 두는 운영자 책임).
- git worktree 통합(`git worktree add/remove`)은 별도 후속 plan — 본 plan은 file system level isolation에 집중.

### Phase 3 — claim 정책 완화
- `internal/store/runs.go::ClaimNextRun` SQL의 `workspace_id` running 제약을 `per_run_worktree=0`인 workspace로만 적용.
- store 테스트: `per_run_worktree=true`인 workspace에서 같은 workspace의 2개 run이 병렬 claim되는지 + `=false`인 workspace는 기존대로 직렬.

### Phase 4 — executor가 worktree cwd 사용
- `internal/app/runtime_executor.go` 또는 worker 측에서 RunContext의 `WorkspaceWorkingDir`을 per_run_worktree 활성 시 새 worktree path로 override.
- worker_store가 run 시작 시 `AllocateRunWorktree`, run 종료 시 cleanup 호출.

### Phase 5 — UI 토글 + 운영 가이드
- `CreateWorkspaceDialog` + Settings `WorkspaceTimeoutRow`에 `per_run_worktree` checkbox. 위험 안내 한 줄(파일 시스템 격리만, git 충돌은 별도 plan).
- README 설정 섹션에 한 단락.

## 제외 범위

- git worktree 통합(`git worktree add/remove`) — 별도 후속 plan. workspace.working_dir이 git repo여야 가능하고, branch / detached HEAD 정책 결정이 필요.
- worktree 파일 자동 시드(workspace.working_dir 파일 복사 등) — agent CLI가 빈 cwd로도 동작한다고 가정. workspace.working_dir 자체는 read 가능한 path로 유지(`--include-directories` 같은 agent 옵션은 호출자 책임).
- per-run worktree와 workspace.WorkingDir의 관계 자동화(예: symlink, bind mount) — 본 plan에서 다루지 않음.
- worktree 용량 / 보존 기간 정책 — 운영자 책임. cleanup만 보장.
