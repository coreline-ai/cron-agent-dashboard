## 개발 목적

`294fe2e` (PM hub chain re-entry exemption) 직후 RFP-2 풀체인 검증에서 드러난 두 결함을 정공으로 해결한다.

1. **결함 #1 — chain prompt invalid UTF-8**
   `internal/store/store.go::capSnapshot`이 byte 단위 4000 절단으로 UTF-8 경계를 무시해 멀티바이트 문자 중간을 끊는다. 그 결과 다음 chain dispatch의 `trigger_content_snapshot`에 invalid byte가 섞이고, codex CLI가 `Failed to read prompt from stdin: input is not valid UTF-8 (invalid byte at offset N)`로 1초 만에 fail한다.
   `internal/worker/executor.go::CapCommentForLogWithStatus`도 truncation 이후에 `firstUTF8Bytes`를 적용해 같은 위험 패턴.

2. **결함 #2 — codex stdout 노이즈 혼입**
   codex CLI가 진단 메시지(예: `MCP issues detected. Run /mcp list for status.`)를 stdout에 흘려서 (a) agent 결과 댓글 본문 첫 줄에 노이즈가 보존되고 (b) 다음 chain dispatch prompt를 추가로 오염시킨다. codex CLI 자체에 stderr 분리 flag 없음.

## 개발 범위

- UTF-8 안전 절단 헬퍼 도입 + `capSnapshot` 전면 교체.
- `CapCommentForLogWithStatus`(또는 호출자)가 truncation **이후**에도 `strings.ToValidUTF8(..., "�")` 보정을 적용.
- chain dispatch가 INSERT하는 `trigger_content_snapshot`이 항상 `utf8.Valid`임을 보장.
- runtime adapter 인터페이스에 stdout sanitize hook 추가 + codex adapter에서 알려진 노이즈 패턴(`MCP issues detected. Run /mcp list for status.` 라인 등) strip.
- executor가 댓글/스냅샷 저장 직전에 adapter sanitize를 적용.
- 모든 Phase는 결함 재현 테스트(failing) → 패치 → 통과 순서(TDD-first).

## 제외 범위
