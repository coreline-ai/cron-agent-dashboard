# Plan 1. Run Lifecycle Hardening

## 목표

`run` 실행 생명주기를 운영 가능한 수준으로 강화한다.

핵심은 아래를 명확히 기록하고 복구하는 것이다.

- 실행 heartbeat.
- 정상/실패/취소/timeout/shutdown/orphan/stale/panic의 구조화된 종료 사유.
- worker panic 시 run이 `running`에 박히지 않게 정리.
- shutdown cancel이 user cancel로 오분류되지 않게 보정.
- startup/self-check에서 stale/orphan 상태를 더 명확히 보고.

## 도입 조건

- 기존 `CompleteRun`의 `WHERE id=? AND status='running'` guard는 유지한다.
- cancel-after-claim race 방지를 위한 `pendingCancels` 정책은 유지한다.
- heartbeat write 빈도는 SQLite 부담을 고려해 기본 10초 이상으로 둔다.
- stale recovery는 active worker가 알고 있는 run을 제외할 수 있어야 한다.

## 데이터 모델 계약

신규 migration 번호는 `0005`를 사용한다.

```sql
ALTER TABLE run ADD COLUMN heartbeat_at TEXT;

ALTER TABLE run ADD COLUMN terminal_reason TEXT NOT NULL DEFAULT ''
  CHECK (terminal_reason IN (
    '',
    'completed',
    'exit_nonzero',
    'timeout',
    'executor_error',
    'worker_panic',
    'claim_preparation_failed',
    'unknown_failure',
    'user_cancelled',
    'issue_cancelled',
    'shutdown',
    'orphan_recovered',
    'stale_recovered'
  ));

ALTER TABLE run ADD COLUMN failure_kind TEXT NOT NULL DEFAULT ''
  CHECK (failure_kind IN (
    '',
    'exit_nonzero',
    'timeout',
    'executor_error',
    'worker_panic',
    'claim_preparation_failed',
    'unknown'
  ));

ALTER TABLE run ADD COLUMN cancel_reason TEXT NOT NULL DEFAULT ''
  CHECK (cancel_reason IN (
    '',
    'user',
    'issue',
    'shutdown',
    'orphan',
    'stale'
  ));

CREATE INDEX idx_run_running_heartbeat
  ON run(heartbeat_at, claimed_at, id)
  WHERE status = 'running';
```

Backfill 정책:

- 기존 `done` → `terminal_reason='completed'`.
- 기존 `failed` → `terminal_reason='unknown_failure'`, `failure_kind='unknown'`.
- 기존 `cancelled`는 `error_message` 기반 best-effort mapping.
- 기존 `running`은 `heartbeat_at=COALESCE(started_at, claimed_at)`.

## 데이터 모델 계약

신규 migration 번호는 `0006`을 사용한다.

```sql
CREATE TABLE run_event (
  id          TEXT PRIMARY KEY,
  run_id      TEXT NOT NULL REFERENCES run(id) ON DELETE CASCADE,
  issue_id    TEXT NOT NULL REFERENCES issue(id) ON DELETE CASCADE,
  seq         INTEGER NOT NULL,
  event_type  TEXT NOT NULL CHECK (
    event_type IN (
      'run_queued',
      'run_claimed',
      'run_prepare_failed',
      'executor_starting',
      'stdout_truncated',
      'cancel_requested',
      'run_cancelled',
      'run_completed',
      'run_failed',
      'orphan_recovered',
      'stale_recovered'
    )
  ),
  severity    TEXT NOT NULL DEFAULT 'info'
              CHECK (severity IN ('debug','info','warn','error')),
  message     TEXT NOT NULL DEFAULT '',
  detail_json TEXT NOT NULL DEFAULT '{}',
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(run_id, seq)
);

CREATE INDEX idx_run_event_run_seq
  ON run_event(run_id, seq);

CREATE INDEX idx_run_event_issue_created
  ON run_event(issue_id, created_at, run_id, seq);

CREATE INDEX idx_run_event_type_created
  ON run_event(event_type, created_at);
```

# Plan 5. Issue Operations Console + Board Filters

## 목표

Board와 IssueDetail을 “작업 조회 화면”이 아니라 “실행 상태를 운영하는 콘솔”로 개선한다.

핵심은 아래다.

- Board에서 `issue.status`뿐 아니라 `execution_status`, agent 기준으로 필터링.
- URL query로 필터 상태 공유.
- IssueDetail에 우측 `IssueSummaryRail` 추가.
- 최신 실패/취소 run 오류를 상단에서 바로 확인.
- action matrix를 명확히 해 실행 취소와 이슈 취소를 혼동하지 않게 한다.

## 도입 조건

- Plan 4의 `StatusPill`, `ConfirmDialog`, `ToastProvider`, `DateTimeText` 선행 권장.
- backend의 `ListIssues`는 이미 `status`, `execution`, `assignee`, `q`, `limit` query를 문서상 지원하나 frontend `useIssuesQuery`는 아직 단순 호출이다.
- MVP에서는 서버 pagination은 제외한다.
- execution filter는 초기에는 client-side로 구현 가능하나, URL query와 API query를 맞추는 방향을 권장한다.

## 예상 영향 파일

| 영역 | 파일 |
|---|---|
| API client | `web/src/api/queries.ts` |
| Board | `web/src/pages/BoardPage.tsx`, 신규 `IssueCard.tsx` 가능 |
| IssueDetail | `web/src/pages/IssueDetailPage.tsx`, 신규 `IssueSummaryRail.tsx` |
| Components | Plan 4 components 사용 |
| CSS | `web/src/styles/global.css` |
| Optional Backend | `internal/store/issues.go`, `internal/httpapi/server.go` — query semantics 점검/보강 |

## API / Query 계약

Frontend query signature를 확장한다.

```ts
type IssuesQueryParams = {
  status?: IssueStatus | 'all';
  execution?: ExecutionStatus | 'all';
  assignee?: string;
  q?: string;
  limit?: number;
};
```

URL 예시:

```text
/w/news/board?status=open&execution=failed&agent=<agent-id>&q=reddit
```

주의:

- 기존 `Issue` type에는 `assignee_agent_id`, `last_run_agent_id`가 빠져 있을 수 있으므로 agent filter 정확도를 위해 type/API scan 확인이 필요하다.
- `store.Issue`는 이미 `AssigneeAgentID`, `LastRunAgentID`를 가지고 있고 JSON으로 내려가므로 TypeScript type만 보강하면 된다.

## 최종 결과 요약

구현 완료 후 기대 상태:

- run은 왜 끝났는지 구조적으로 설명 가능하다.
- 실행 중 heartbeat/stale/panic/shutdown이 관측 및 복구된다.
- run별 기술 이벤트 timeline으로 debugging이 쉬워진다.
- Autopilot은 실패가 UI에 남고 마지막 생성 이슈로 이동 가능하다.
- 프론트엔드는 상태/확인/피드백이 일관된다.
- Board와 IssueDetail은 실제 agent 작업 운영 콘솔로 동작한다.

## 잔여 리스크 / 후속 과제

- `execution` 필터의 SQL화는 Plan 5에서 문서화만 하고, 성능 문제가 보이면 별도 PR로 분리한다.
- MCP server, artifact/attachment, per-run worktree는 이번 5개 계획 완료 후 재평가한다.
- 자동 log retention/maintenance scheduler는 Run Lifecycle 안정화 이후 별도 계획으로 진행한다.
