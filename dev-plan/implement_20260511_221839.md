작성 일시: `2026-05-11 22:18:39 KST`

이 문서는 `Corn Agent Dashboard` 저장소의 현재 MD 문서를 상세 분석한 뒤, `Corn Agent Dashboard` 최초 구현 범위를 고정하고 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적

- `Corn Agent Dashboard`를 문서 설계에서 실제 동작 가능한 MVP로 구현한다.
- 기존 Corn Design Reference의 검증된 핵심 UX인 **보드 / 댓글 / 에이전트 멘션 위임 / Autopilot**만 단일 사용자 로컬 도구로 재구성한다.
- 배포 단위는 **단일 Go 바이너리 + SQLite 파일 + Vite SPA embed**로 제한한다.
- 구현 중 상태 모델, 큐, 멘션, stdout 처리, identifier routing이 문서와 어긋나지 않도록 Phase별 자체 테스트로 검증한다.

## 현재 구현/문서 분석 요약

- 문서 정합성 이슈는 1차 수정 완료했다.
  - `issue.status = open | done | cancelled`와 `run.status = queued | running | done | failed | cancelled` 분리 원칙을 PRD/TRD/ARCHITECTURE/ROADMAP에 맞췄다.
  - mention은 assignee를 바꾸지 않고 새 `run.agent_id`만 멘션 agent로 설정한다는 정책을 유지했다.
  - stdout은 run log 10MB cap + comment 64KB cap + log link 전략으로 정리했다.
- 병렬 에이전트 개발 결과로 초기 foundation 구현이 반영되었다.
  - Go module path: `github.com/coreline-ai/corn-agent-dashboard`
  - Binary/package path: `cmd/corn-agent-dashboard`
  - SQLite migration/store/API skeleton, worker/runtime/scheduler skeleton, Vite React 7-route skeleton 추가
- 현재는 “릴리스 완료 MVP”가 아니라 “빌드/테스트 가능한 초기 통합 스캐폴드” 단계다.
  - 완료: DB/API foundation, durable queue claim logic, mention dispatch, runtime executor primitives, scheduler wrapper, frontend route skeleton
  - 후속: worker pool ↔ store/main wiring, scheduler DB reload wiring, Go `embed.FS` static serving, CLI backup/restore, 실제 CLI runtime smoke, release packaging
- 문서/바이너리 이름은 `corn-agent-dashboard`, GitHub repository URL은 `https://github.com/coreline-ai/corn-agent-dashboard.git`로 기록한다.

## 문서 기반 제약 사항

- `issue.status`는 사용자 의도만 표현한다.
  - 허용값: `open | done | cancelled`
  - run이 running/failed/cancelled가 되어도 issue.status를 자동으로 `running`/`failed`로 바꾸지 않는다.
- `run.status`는 실행 상태만 표현한다.
  - 허용값: `queued | running | done | failed | cancelled`
- `execution_status`는 저장하지 않고 API 응답에서 계산한다.
  - running run이 있으면 `running`
  - 아니면 queued run이 있으면 `queued`
  - 아니면 최신 종료 run의 status
  - run이 없으면 `idle`
- mention은 issue assignee를 바꾸지 않는다.
  - `run.agent_id`만 mentioned agent로 설정한다.
- MVP는 workspace당 running run 1개만 허용한다.
  - 같은 `workspace.working_dir`에서 파일 충돌을 막기 위한 안전 장치다.
- stdout은 두 단계로 제한한다.
  - run log file: 10MB cap, cap 이후에도 pipe drain
  - comment content: 64KB cap, 초과 시 log link 안내
- markdown rendering은 raw HTML을 허용하지 않는다.
  - `react-markdown` + `remark-gfm`
  - `rehype-raw` 금지
- issue URL은 slug + identifier를 지원해야 한다.
  - `/w/:slug/issues/:identifier`
  - API는 workspace context에서 UUID 또는 identifier를 모두 resolve한다.
- external bind(`0.0.0.0`) 시 token 미설정이면 부팅 거부한다.

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- `html-news-creator`는 참조 전용이며 수정하지 않는다.
- 단순함 우선: 단일 사용자 MVP에 필요 없는 추상화는 추가하지 않는다.
- 모든 DB write는 짧은 transaction으로 유지한다.
- 모든 handler write path는 raw path param을 그대로 DB update/delete에 넣지 않고 loader/validator를 통과시킨다.
- Phase별 self-test가 실패하면 다음 Phase로 넘어가지 않는다.

## 잔여 리스크 / 후속 과제

- `corn-agent-dashboard` 문서명과 `corn-agent-dashboard` 원격 저장소명의 불일치는 릴리스 전에 반드시 결정해야 한다.
- 실제 `codex`, `claude`, `gemini` CLI의 non-interactive 실행 옵션은 설치 버전별 차이가 있을 수 있으므로 fake adapter tests와 별도로 수동 smoke test가 필요하다.
- `modernc.org/sqlite`와 partial index / `RETURNING` / CTE `UPDATE ... RETURNING` 지원 범위는 Phase 1에서 실제 테스트로 검증해야 한다.
- SQLite write lock은 단일 사용자 기준 충분하지만, worker + scheduler + HTTP write가 겹칠 때 retry 정책이 필요할 수 있다.
- stdout comment 64KB cap은 UX상 충분한지 실제 agent 결과로 조정할 수 있다.
- workspace-level serial execution은 안전하지만 처리량이 낮다. Phase 2에서 per-run worktree를 도입하면 병렬 실행을 다시 검토한다.
