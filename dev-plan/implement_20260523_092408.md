## 개발 목적

`README.md` 의 코드 레벨 전문가 분석 요약에 명시된 5개 후보:

1. realtime streaming (SSE/WebSocket) 도입 여부
2. Homebrew tap 실제 publish
3. workspace history import 복원
4. per-run worktree 디스크 사용량 관측/GC
5. e2e-full CI 승격

모두 채택해 한 사이클로 마무리한다. realtime streaming은 양자택일 결정이 필요하므로 SSE(서버 보낸 이벤트)로 고정 — 단방향 + 표준 HTTP + 기존 chi 라우터 그대로 사용 가능.

## 개발 범위

- **Track A — per-run worktree 디스크 사용량 관측 + GC**: maintenance runner가 매 tick마다 `<data_dir>/worktrees/` 디스크 사용량을 측정해 `system_state` KV에 기록. 모든 run이 terminal인 디렉터리를 prune. 결과는 Settings UI에 노출.
- **Track B — e2e-full CI 승격**: 현재 `make e2e-smoke`만 CI에서 돈다. `e2e-full`도 별도 job으로 추가(병렬, 실패 시 release 차단은 안 함 — 안정화 단계).
- **Track C — workspace history import 복원**: v2 export의 issues/comments/runs/attachments 슬라이스를 새 워크스페이스에 복원. `ImportOptions.IncludeHistory bool` 추가.
- **Track D — realtime streaming (SSE)**: `/api/issues/{id}/events` SSE 엔드포인트. 새 run_event가 추가되면 push. IssueDetailPage가 EventSource로 구독.
- **Track E — Homebrew tap publish CI step**: 새 GitHub Actions workflow가 release publish 후 formula 템플릿을 채워 tap repo로 PR 생성. tap repo가 없으면 step skip.

## 제외 범위

- WebSocket — SSE로 충분(단방향).
- Workspace UI의 streaming dashboard — 본 plan은 IssueDetail에 한정.
- attachment GC — 별도 plan (현재는 row만 cleanup, 파일은 RemoveAttachmentFile로 즉시 제거되므로 불필요).
