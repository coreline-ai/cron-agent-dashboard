## 개발 목적

남은 backlog 순차 진행 — 세 번째 항목 **workspace import/export**. fresh clone에서 운영 환경을 옮기거나, 잘 짜인 workspace 설정을 다른 환경에 복제할 수 있는 기본 mechanism을 제공한다. 본 plan은 **운영 설정 단위**(workspace + agents + skills + autopilot rules)만 다룬다. **이슈/run/comment 데이터는 본 plan 범위 밖** — 별도 plan으로 분리(데이터 크기 + 민감정보 마스킹 정책이 더 무거움).

## 개발 범위

- `internal/app/workspace_io.go` 신설 — `WorkspaceExport` 직렬화 구조 + `ExportWorkspace(ctx, st, slug)` + `ImportWorkspace(ctx, st, export, opts)`.
- 데이터: workspace 메타 / 모든 agents(main + worker) / skills registry + agent_skill 할당 / autopilot rules. 이슈/run/comment/run_event는 제외.
- 직렬화 ID는 export에 포함하되 import 시 무시(새 ID 생성). instructions_version도 1부터 재시작.
- import 옵션: 같은 slug 충돌 시 정책 `skip` / `rename`(`<slug>-2` 추가) / `error`(기본). 본 plan은 `error`만 우선 — 향후 옵션 확장.
- CLI: `cron-agent-dashboard workspace-export --slug <s> --to <file.json>` + `workspace-import --from <file.json>` 명령. 기존 `backup`/`restore`는 SQLite 파일 단위라 의미 다름 — 별도 명령으로 분리.
- 테스트: export → import 라운드트립이 동일 구조의 workspace를 만들어내는지 + slug 충돌 시 ErrConflict.

## 제외 범위

- 이슈/run/comment/run_event export — 데이터 크기 + 민감정보 마스킹 정책. 별도 plan.
- HTTP `/api/workspaces/{slug}/export` 라우트 — CLI 우선, API는 후속.
- 다중 workspace 일괄 export — 단일 workspace만.
- import 시 token / secret 마스킹 정책 — instructions에 비밀이 들어 있으면 운영자 책임 (README 보안 섹션 인용).
- demo seed와 통합 — `seed example` 명령은 그대로 별개 (코드 path 다름).
