## 개발 목적

backlog 순차 진행 — 다섯 번째 항목 **external webhook**. 워크스페이스에서 일어나는 run/issue 종료 이벤트를 외부 시스템(Slack/Discord 라우터, 사내 알림 서버, 자체 자동화)으로 발신할 수 있게 한다. MVP는 **워크스페이스 단위 구독 + HMAC-SHA256 서명 + 단순 retry + delivery 감사 로그**까지.

## 개발 범위

| Phase | 내용 |
|---|---|
| 1 | schema(migration `0019`) + 모델: `webhook` / `webhook_delivery` 테이블, 모델 struct |
| 2 | store CRUD: `CreateWebhook` / `ListWebhooks` / `GetWebhook` / `UpdateWebhook` / `DeleteWebhook` + delivery 조회 |
| 3 | lifecycle hook: run 완료/실패, issue done/cancelled 시 `EnqueueWebhookDeliveries` 호출 → `webhook_delivery` `pending` row 생성 |
| 4 | HTTP API: `GET/POST/PUT/DELETE /api/workspaces/:slug/webhooks`, `GET /api/webhooks/:id/deliveries` |
| 5 | background dispatcher: pending delivery를 polling → HTTP POST + HMAC 서명 + 결과 기록. retry는 1회(즉시 + 30초 backoff), 그 후 `failed` |
| 6 | UI: workspace 설정 페이지에 webhook 목록/추가/삭제, recent deliveries 표시 |

## 제외 범위

- 글로벌(non-workspace) webhook — 본 plan은 workspace-scoped만.
- 이벤트 타입 확장: 본 plan은 `run.completed`(성공), `run.failed`, `issue.done`, `issue.cancelled` 4종으로 한정. 다른 종류는 후속.
- 다회 retry (3회+ exponential backoff), dead-letter queue — 본 plan은 1회 retry.
- 외부 시스템 연동(Slack incoming webhook payload 변환 등) — 본 plan은 raw JSON payload만.
- WebSocket / SSE 전달, mTLS — HTTP POST + HMAC만.
- delivery payload 마스킹/필터 — instructions, comment body 등 민감 정보가 그대로 들어갈 수 있음. README 보안 단락에 명시만.

## Phase 5. background dispatcher + HMAC

### 목표

- pending delivery를 폴링 → HTTP POST + HMAC-SHA256 서명 → 결과 기록.

### 구현 태스크

- [ ] `internal/app/webhook_dispatcher.go` — `WebhookDispatcher` struct + `Start(ctx)` / `Stop(ctx)`.
- [ ] 폴링 간격 30초(서버 설정으로 조정 가능, 본 plan은 hard-coded). 작업: pending 중 `next_attempt_at <= now` 한 건 fetch → `UPDATE ... SET status=...` row-level lock.
- [ ] HTTP POST timeout 10초. body는 `payload_json`. 헤더: `Content-Type: application/json`, `X-Cron-Agent-Event: <event_type>`, `X-Cron-Agent-Delivery: <delivery_id>`, secret이 있으면 `X-Cron-Agent-Signature: sha256=<hex>`.
- [ ] 결과:
  - 2xx → `delivered`, `delivered_at`, `status_code`.
  - 그 외 → attempt < 1 이면 `next_attempt_at = now + 30s` (1회 retry), `attempt += 1`. attempt >= 1 이면 `failed`.
- [ ] `cmd/cron-agent-dashboard/main.go::serve`에 dispatcher 기동/정지 연결.

## 잔여 리스크 / 후속 과제 (본 plan 범위 밖)

- **delivery payload masking / redaction** — instructions / comment body가 그대로 전송된다. 사용자가 정책을 옵션화하려면 별도 plan.
- **다회 retry + exponential backoff** — 본 plan은 1회 retry. SaaS급 안정성은 별도 plan.
- **이벤트 종류 확장** — autopilot triggered, run started, mention dispatched 등은 후속.
- **글로벌 / 시스템 webhook** — workspace-scoped만. 시스템 audit / heartbeat 발신은 별도 plan.
- **delivery 결과 알림 UI 강화** — 실패 누적 시 자동 비활성, 표시 dashboard 등.
