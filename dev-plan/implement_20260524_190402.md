# implement_20260524_190402.md

작성 일시: `2026-05-24 19:04:02 KST`

이 문서는 **5개 이상 워크스페이스 + 멀티 에이전트**를 실제 개발 테스트에 투입하기 위한 전체 실행 계획이다. 범위는 Cron Agent Dashboard 자체를 테스트베드로 삼아, 여러 워크스페이스가 병렬 또는 리더 지시 방식으로 안전하게 작업하고 결과를 통합할 수 있는지 검증하는 데 맞춘다.

## 개발 목적

- Cron Agent Dashboard가 여러 워크스페이스/에이전트/런을 동시에 운영할 때 실무 개발 흐름을 버틸 수 있는지 검증한다.
- 각 워크스페이스에 독립 프로젝트 1개씩을 할당해 병렬 개발, 리더 지시, 멘션 위임, run feed, chain dashboard, worktree GC, import/export, release dry-run까지 실제로 사용한다.
- 테스트 종료 후 코드/문서/운영 UX에서 발견한 개선점을 후속 plan으로 분리할 수 있게 결과를 수집한다.

## 개발 범위

- 상위 리더 워크스페이스 1개 + 실행 워크스페이스 6개, 총 7개 워크스페이스를 구성한다.
- 각 실행 워크스페이스는 main lead agent 1개 + worker agent 2~3개를 가진다.
- 각 워크스페이스에는 충돌이 적은 독립 프로젝트 1개를 할당한다.
- 병렬 모드와 리더 지시 모드를 모두 검증한다.
- 각 워크스페이스는 별도 git branch/worktree와 파일 소유 경계를 가진다.
- 모든 결과는 self-test, global integration test, run/event 로그, token/cost, worktree disk usage로 검증한다.

## 제외 범위

- 실제 외부 Homebrew tap publish, GitHub release 배포, production secret 사용은 제외한다. release/tap은 dry-run까지만 수행한다.
- 무제한 auto-chain, 자동 merge, force push, main 직접 수정은 제외한다.
- 외부 유료 API/비공개 저장소를 필수 조건으로 두지 않는다.
- 워크스페이스 간 네이티브 cross-workspace issue linking이 현재 구현되지 않았으므로, 리더 지시는 API/수동 issue 생성과 링크 기반으로 시뮬레이션한다.

## 리더 지시/병렬 실행 운영 방식

### 병렬 모드

1. `pm-command-hub`가 6개 실행 workspace에 같은 시작 시각으로 seed issue를 만든다.
2. 각 workspace의 `@<Area>Lead`가 자기 프로젝트를 2~4개 하위 태스크로 쪼갠다.
3. worker agent는 파일 소유 경계 내에서만 수정한다.
4. 리더는 `/w/:slug/runs`와 chain dashboard를 보며 queued/running/failed를 모니터링한다.
5. 실패 run은 workspace lead가 원인 분석 후 1회 재실행하거나 scope를 축소한다.

### 리더 지시 모드

1. `@ProgramLead`가 결과물을 리뷰하고 통합 위험을 분류한다.
2. 수정 요청은 대상 workspace issue 댓글에 명확한 acceptance criteria로 남긴다.
3. workspace lead가 `@WorkerName` 멘션으로 재작업을 위임한다.
4. 재작업 완료 후 `@Integrator`가 diff/test 결과를 보고 통합 순서를 조정한다.
5. 최종 merge 후보는 integration branch에서만 합친다.

## 성공 기준

- 7개 워크스페이스가 생성되고, 6개 실행 워크스페이스가 각각 독립 프로젝트를 완료 또는 명확한 실패 사유로 종료한다.
- 병렬 run 3개 이상이 동시에 실행되는 동안 dashboard/API가 안정적으로 동작한다.
- 리더 지시형 재작업이 최소 1회 이상 실제 run으로 검증된다.
- per-run worktree가 병렬 실험 후에도 안전하게 GC되고 queued/running worktree가 삭제되지 않는다.
- export/import, SSE, release dry-run, screenshot/demo 갱신 중 최소 3개 이상의 운영 기능이 실사용 경로로 검증된다.
- 최종 integration branch가 필수 테스트를 통과하거나, 실패 시 재현 가능한 blocker 목록을 남긴다.
