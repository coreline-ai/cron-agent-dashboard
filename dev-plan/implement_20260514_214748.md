# implement_20260514_214748.md

작성 일시: `2026-05-14 21:47:48 KST`

이 문서는 9차 코드 레벨 리뷰에서 확인된 잔여 안정성/운영성 항목을 실제 구현 범위로 고정하고, 병렬 에이전트 작업 중 충돌을 줄이기 위한 작업 문서다.

## 개발 목적
- 9차 리뷰의 단기 권장 항목을 릴리스 전 안전 보강으로 반영한다.
- 이미 완료된 prompt safety / trigger snapshot / explicit-only chaining 정책은 유지하고, 이번 작업에서는 런타임 lifecycle, audit event, autopilot failure 정책을 보강한다.
- 59개 uncommitted 상태는 되돌리지 않고, 이번 변경만 의미 단위로 추가한다.

## 개발 범위
- Run cancel/lifecycle hardening
  - fragile string 기반 cancel reason 분류 의존도를 낮춘다.
  - claim 준비 실패 시 run 실패 기록 오류를 무시하지 않는다.
- Run event/audit hardening
  - `run_event.detail_json`에 안전한 크기 제한을 둔다.
  - `appendRunEventTx`의 불필요한 `run -> issue_id` 조회를 줄인다.
- Autopilot failure policy
  - 연속 실패가 임계치 이상이면 rule을 자동 비활성화한다.
  - UI/API가 기존 failure visibility와 일관되게 동작하도록 한다.
- 운영 로그 기본값
  - main process의 `slog.Default()`가 JSON handler를 사용하도록 초기화한다.
- 검증
  - 각 범위별 회귀 테스트 추가 후 `go test ./...`, `go vet ./...`, `pnpm --filter web build` 실행.

## 제외 범위
- stale recovery zombie process kill 구현은 이번 범위에서 제외한다. OS/process ownership 설계가 필요하므로 별도 Phase로 남긴다.
- worker_panic 자동 차단 정책은 제외한다. 재시도/차단 정책과 함께 설계한다.
- frontend unit test stack(vitest/react-testing-library) 도입은 제외한다. 별도 frontend testing plan으로 분리한다.
- 대량 uncommitted 파일 commit/push는 제외한다. 사용자가 별도 요청하면 ship 단계에서 처리한다.
- 자동 chain 구현은 제외한다. 현재 `docs/CHAINING.md`의 explicit-only 정책을 유지한다.
