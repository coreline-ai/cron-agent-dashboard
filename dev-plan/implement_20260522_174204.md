backlog 순차 진행 — 여섯 번째 항목 **첨부 파일(issue attachment)**. RFP / 운영 워크플로우에서 brief / 견적서 / 스크린샷 / 로그 등을 이슈에 함께 보관할 수 있게 한다. MVP는 이슈 단위 attachment + 단순 파일 시스템 storage + 보안 안내.

## 제외 범위

- comment 첨부 — 본 plan은 issue 첨부만. comment 첨부는 별도 plan(payload/UI 결합도 다름).
- 이미지 inline preview / thumbnail 생성 — `Content-Type`을 헤더로만 반환, 브라우저가 처리.
- 외부 storage(S3, MinIO) — 본 plan은 로컬 파일 시스템만.
- MIME deep-sniff / 바이러스 스캔 / EXIF 제거 — 본 plan은 사이즈/이름 검증만. 보안 노트 README에 명시.
- 멀티파일 동시 업로드 — 본 plan은 단일 파일 업로드만. 사용자가 N번 반복.
- attachment audit log / 다운로드 로그 — 후속.
- backup / export에 attachment 포함 — 후속.

## 파일 시스템 layout

- `<data_dir>/attachments/<attachment_id>` (flat). 메타데이터는 DB에만. 운영자가 디스크에서 파일명을 봐도 ID밖에 안 보이므로 PII 누출 방어 효과.
- 디렉토리 권한 `0700`, 파일 권한 `0600`.

## 사이즈 / 검증 정책

- 최대 파일 크기: **10 MB** (`AttachmentMaxBytes = 10 * 1024 * 1024`). HTTP request body cap도 같은 값으로 강제.
- 파일명: `filename`은 user-provided. SQL/path injection 방지 위해 storage 경로엔 attachment ID만 사용하고 filename은 DB에만.
- Content-Type: 클라이언트 제공값을 그대로 신뢰(deep-sniff 없음). 단 빈값/공백은 `application/octet-stream`으로 normalize.
- SHA256: 업로드 시 계산해 DB에 저장. 중복 업로드 검출은 향후 활용.
